# 健身管理系统 - 管理后台数据流与需求梳理

## 一、项目概述

### 1.1 技术栈
- **后端**: Spring Boot 4.0.4 + JPA + MySQL
- **前端**: Taro (小程序) + UI-Origin (管理后台原型)
- **Java 版本**: 17

### 1.2 系统角色
```
UserRole 枚举:
- CLIENT    - 客户端用户
- COACH     - 教练端用户
- ADMIN     - 管理后台用户
```

## 三、数据流与业务流程

### 3.1 核心业务流程图

```
┌─────────────────────────────────────────────────────────────────┐
│                        管理后台功能模块                          │
├─────────────┬─────────────┬─────────────┬─────────────────────┤
│   用户管理   │   教练管理   │   商品管理   │      订单管理        │
│ - 用户列表   │ - 教练列表   │ - 套餐管理   │ - 课程订单管理       │
│ - 会员档案   │ - 教练审核   │ - 商品管理   │ - 商品订单管理       │
│ - 角色分配   │ - 排班管理   │ - 库存管理   │ - 退款处理          │
│ - 积分管理   │ - 业绩查看   │ - 上下架    │ - 物流管理          │
├─────────────┴─────────────┴─────────────┴─────────────────────┤
│                        数据监控模块                             │
│ - 预约记录监控  - 核销记录查询  - 体测数据查看  - 通知管理      │
│ - 轮播图配置    - 问卷数据查看  - 购物车分析   - 运营统计      │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 预约流程数据流

```
用户端发起预约
    │
    ▼
POST /api/bookings
    │
    ▼
校验：时段可用性 + 用户课时 + 冲突检测
    │
    ▼
创建 Booking 记录
    │
    ├──▶ 更新 CoachScheduleSlot (标记已预约)
    │
    ├──▶ 扣减 CourseOrder.remainingSessions
    │
    └──▶ 创建 NotificationItem (通知教练)
            │
            ▼
        教练端收到通知
```

**管理后台监控点**:
- 预约状态变更记录
- 异常预约处理 (冲突、超时)
- 教练代客预约审核

---

### 3.3 套餐购买流程数据流

```
用户选择套餐
    │
    ▼
POST /api/checkout/preview (预览)
    │
    ├──▶ 读取 PackageProduct
    │
    ├──▶ 读取 ClientProfile.points (积分)
    │
    └──▶ 计算：价格 - 积分抵扣 = 实付
            │
            ▼
POST /api/orders/course (创建订单)
            │
            ▼
创建 CourseOrder (status=ACTIVE)
            │
            ├──▶ 更新 ClientProfile.points (增加奖励积分)
            │
            └──▶ 创建 Payment (支付单)
                    │
                    ▼
                支付成功回调
                    │
                    ▼
                订单生效，课时可用
```

**管理后台操作点**:
- 套餐上下架 (PackageProduct.saleStatus)
- 价格调整
- 订单退款 (CourseOrder.status -> REFUNDING)
- 课时调整 (手动修正 usedSessions/remainingSessions)

---

### 3.4 商城购买流程数据流

```
用户添加购物车
    │
    ▼
POST /api/cart/items
    │
    ├──▶ 创建/更新 CartItem
    │
    └──▶ 读取 Product.stock (库存校验)
            │
            ▼
用户结算
    │
    ▼
POST /api/checkout/preview
    │
    ├──▶ 校验库存
    │
    ├──▶ 计算积分抵扣
    │
    └──▶ 返回实付金额
            │
            ▼
POST /api/orders/product
    │
    ├──▶ 创建 ProductOrder
    │
    ├──▶ 创建 ProductOrderItem 列表
    │
    ├──▶ 扣减 Product.stock
    │
    └──▶ 清空 CartItem
            │
            ▼
        发货流程
            │
            ▼
        更新 trackingNumber
```

**管理后台操作点**:
- 商品上下架
- 库存管理
- 订单发货 (更新 trackingNumber)
- 退款处理

---

### 3.5 教练排班与授课流程数据流

```
教练设置排班
    │
    ▼
创建 CoachScheduleSlot 列表
    │
    ├──▶ 设置 available=true
    │
    └──▶ 配置 roomName
            │
            ▼
用户预约时段
    │
    ▼
更新 CoachScheduleSlot
    │
    ├──▶ available=false
    │
    └──▶ bookingId=预约 ID
            │
            ▼
课程完成
    │
    ▼
创建 CoachClass 记录
    │
    ├──▶ 记录学员信息
    │
    ├──▶ 记录训练内容
    │
    └──▶ status=COMPLETED
            │
            ▼
        更新 CourseOrder.usedSessions
```

**管理后台监控点**:
- 教练排班合规性
- 课程完成率统计
- 教练业绩统计 (classCount, rating)

---

```
            └──▶ 更新 attendance
                    │
                    ▼
                客户端查看身体趋势
```

**管理后台操作点**:
- 体测数据查看

### 3.7 扫码核销流程数据流

```

    │
    ▼

    │

POST /api/coach/checkin/scan
    │

            │

```

## 八、注意事项

### 8.1 数据一致性

## 九、总结

本文档梳理了健身管理系统的完整数据结构和管理后台需求:

- **共 24 个数据实体**,涵盖用户、教练、预约、商品、订单、体测、通知等模块
- **7 大核心业务流程**,包括预约、购买、授课、核销等
- **40+ 个管理后台 API**,覆盖用户管理、教练管理、商品管理、订单管理等
- **8 个管理后台页面模块**,建议按优先级分阶段开发

管理后台的核心目标是:
1. 维护基础数据 (用户、教练、商品)
2. 监控业务流程 (预约、订单、核销)
3. 提供运营能力 (配置、通知、统计)
4. 处理异常情况 (退款、纠纷、数据修正)

建议按 P0→P1→P2→P3 的优先级逐步实现，优先打通基础管理功能，再补充数据统计和高级功能。
